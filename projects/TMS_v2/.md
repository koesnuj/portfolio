# TestRail 대체 시스템 개발 기록

개발 경험이 없는 QA 엔지니어가 Cursor AI를 활용하여 일주일 만에 테스트 관리 시스템을 구축한 과정

---

## 배경

TestRail의 높은 라이선스 비용이 부담스러워 자체 시스템을 만들어보기로 했다. 문제는 코딩 경험이 전혀 없다는 것이었다. Cursor AI를 알게 되어 반신반의하며 시도해봤다.

---

## 개발 일지

### Day 1: 프로젝트 초기화

AI에게 "TestRail 같은 테스트 관리 시스템을 만들고 싶다"고 말했더니 Node.js, Express, React로 프로젝트 구조를 잡아줬다. 솔직히 이 기술들이 뭔지 잘 몰랐다.

터미널에 `npm run dev`를 입력했더니 브라우저에 로그인 화면이 떴다. 아무것도 안 했는데 뭔가 작동하는 게 신기했다.

**학습한 개념**
- `npm run dev`: 개발 서버 실행 명령어
- `localhost:5173`: 로컬 개발 서버 주소
- 백엔드/프론트엔드 분리 구조

### Day 2: 데이터 구조 설계

TestRail처럼 폴더로 테스트 케이스를 정리하고 싶었다. AI가 Prisma로 계층형 구조를 만들어줬다.
```prisma
model Folder {
  id        String   @id
  name      String
  parentId  String?
  children  Folder[]
}
```

`parentId`를 보니 대충 이해가 됐다. 상위 폴더를 가리키는 거구나.

**첫 에러**

폴더 만들기 버튼을 눌렀는데 아무 반응이 없었다. F12를 눌러 콘솔을 보니 "CORS policy" 어쩌고 빨간 글씨가 있었다. 에러 메시지를 AI에게 복사해서 보냈더니 바로 해결해줬다. 에러가 나도 당황하지 않아도 되는구나 싶었다.

### Day 3: 드래그앤드롭 구현

기본 기능이 작동하니 욕심이 생겼다. TestRail처럼 드래그로 폴더를 이동하고 싶었다. AI가 @dnd-kit을 사용해서 구현해줬는데, 실제로 폴더가 움직이는 걸 보니 좀 놀라웠다.

**예외 처리**

테스트해보니 폴더를 자기 자신 안으로 넣을 수 있는 버그가 있었다. AI에게 말했더니 순환 참조 방지 로직을 추가해줬다. QA 일하면서 생긴 습관이 도움이 됐다.

### Day 4: UI 개선

기능은 되는데 화면이 너무 밋밋했다. TestRail처럼 왼쪽에 폴더 트리, 가운데 케이스 목록, 오른쪽에 상세 정보를 보여주고 싶었다.

AI가 Tailwind CSS로 레이아웃을 만들어줬다. `flex`, `gap-4`, `bg-white` 같은 클래스명을 보니 영어 단어 조합이라 이해하기 쉬웠다. 코딩이라기보다 레고 조립 같은 느낌이었다.

### Day 5: 테스트 플랜 기능

테스트 케이스를 모아서 플랜을 만들고 Pass/Fail/Block 결과를 기록하는 기능이 필요했다. Plan과 PlanItem 모델을 만들어달라고 했다.

**상태 관리 문제**

여기서 좀 막혔다. 드롭다운에서 Pass를 선택하면 화면에 반영되고 서버에도 저장되어야 하는데, 한쪽만 되거나 아예 안 되는 경우가 많았다.

- "저장은 되는데 화면이 안 바뀐다"
- "화면은 바뀌는데 새로고침하면 원래대로 돌아간다"

이런 문제를 하나씩 AI에게 설명하면서 고쳐나갔다. 한 번에 되는 법이 없더라.

## 구현된 기능

일주일 만에 생각보다 많은 게 완성됐다.

- 회원가입/로그인 (관리자 승인)
- 계층형 폴더 구조
- 드래그앤드롭 정렬
- 테스트 플랜 생성 및 실행
- Pass/Fail/Block 결과 기록
- 통계 대시보드
- PDF/Excel 내보내기
- 반응형 UI

## AI 활용 팁

### 1. 에러 메시지는 그대로 복사

에러가 나면 전체 메시지를 복사해서 AI에게 보낸다. 영어여도 상관없다. AI가 알아서 해석하고 고쳐준다.

### 2. 작게 시작해서 키워가기

처음부터 "TestRail 똑같이 만들어줘"는 안 통한다. 로그인 → 테스트 케이스 CRUD → 폴더 구조 → 드래그앤드롭 순서로 하나씩 추가했다.

### 3. 직접 써보면서 피드백

QA 하듯이 직접 테스트하면서 불편한 점을 찾아 개선했다. "버튼이 너무 작아서 클릭하기 힘들다" 같은 구체적인 피드백이 효과적이었다.

### 4. 모르는 건 바로 질문

"useState가 뭐야?" 같은 기초적인 질문도 했다. AI는 판단하지 않으니 부담 없이 물어볼 수 있다.

## 회고

### 좋았던 점

진입 장벽이 낮았다. 코딩을 몰라도 시작할 수 있었고, 바로바로 결과를 확인할 수 있어서 재미있었다. 하다 보니 코드가 조금씩 눈에 들어오는 것도 나쁘지 않았다.

### 힘들었던 점

디버깅에 시간이 많이 걸렸다. 특히 상태 관리 같은 개념은 이해하는 데 좀 걸렸다. AI가 가끔 다른 스타일로 코드를 생성해서 일관성이 떨어질 때도 있었다.

### 배운 것

코딩을 "배웠다"고 하긴 애매하지만, Git, REST API, React 기초, SQL 같은 개념은 대충 이해하게 됐다. 최소한 개발자와 대화할 수는 있을 것 같다.
