## 🚀 시작하게 된 계기

이전 회사에서는 약 5년 동안 TestRail을 사용했습니다. 기능도 안정적이고 강력했죠.  
하지만 이직 후 현재 회사에서는 **Google 스프레드시트**로 테스트 케이스를 관리하고 있었습니다.

### 스프레드시트의 한계

스프레드시트는 가볍고 접근성은 좋지만, 명확한 한계가 있었습니다:

- ❌ 버전 관리가 어렵고
- ❌ 동시 작업 시 충돌이 발생하며
- ❌ 테스트 케이스가 많아지면 구조가 쉽게 무너짐

### 대안 탐색의 실패

여러 저가형 테스트 관리 툴도 시도해봤지만, 다음과 같은 이유로 결국 다시 스프레드시트로 되돌아오곤 했습니다:

- 기능 부족
- UX 불편
- 우리 팀의 사용 패턴과 불일치

### 프로젝트 결심

그러던 중 문득 떠올렸습니다.

> *"차라리 우리 팀에 딱 맞는 테스트 관리 시스템을 직접 만들면 어떨까?"*

마침 Cursor AI 기반의 "바이브 코딩"이 주목받던 시기였고,  
**"나도 한번 해볼 수 있겠다"**는 생각으로 프로젝트를 시작했습니다.

## 🔥 프로젝트 시작 — 비전공자의 도전

### 나의 개발 경험

제가 가진 개발 경험은 매우 제한적이었습니다:

- ❌ React 경험 없음
- ❌ 프론트엔드 프레임워크 이해도 낮음
- ✅ Python으로 간단한 스크립트 작성 정도

### 첫 번째 프롬프트

Cursor에 처음 요청했던 것은 단순히:

> *"테스트 관리 시스템을 만들고 싶어요."*

Cursor는 **React + Node.js + Express** 기술 스택을 제안했고,  
저는 그대로 프로젝트 기반으로 삼아 개발을 시작했습니다.

### AI 중심 개발

이 프로젝트는 **100% Cursor 중심의 AI 개발 프로젝트**였고,  
기술 스택부터 구조 설계까지 대부분 AI의 제안을 기반으로 진행했습니다.

## ⚙️ 개발 과정 — 감탄에서 한계까지

### 1️⃣ 초반: "와, 이게 진짜 되네?"

기본적인 UI와 기능들이 순식간에 만들어지는 모습을 보며 AI의 잠재력을 강하게 체감했습니다.

- ✅ 로그인
- ✅ 리스트 화면
- ✅ CRUD 기능 구현

**"사람들이 왜 AI 개발에 열광하는지"** 몸으로 느꼈습니다.

### 2️⃣ 기능이 쌓이면서 드러난 문제점

#### 🚨 AI는 모든 걸 기억하지 않는다

- 기존 기능을 망가뜨리는 경우 발생
- 불필요하다고 판단한 코드를 임의로 수정
- 맥락이 사라지면 전혀 다른 코드를 제안

이로 인해 **"집에 가고 싶다…"**는 순간도 많았습니다.

#### 🚨 프롬프트가 모호하면 결과도 모호하다

**나쁜 예:**
> *"이런 느낌으로 만들어줘"*  
> → 그런 느낌이지만 구조가 엉망인 코드가 생성됨

**문제점:**
- 비일관적 상태 관리
- 복잡한 로직
- 재사용성 떨어짐

**깨달음:**  
문제 대부분은 **애매한 요구사항 전달**에서 비롯됨을 깨달았습니다.

## 🧩 개발 방식 — 기능 중심 설계

개발 언어와 구조를 완전히 이해하지 못한 상태에서  
제가 집중한 것은 **'기능을 정확하게 정의하는 것'** 뿐이었습니다.

### 핵심 질문들

제가 스스로 던진 질문들:

- 💭 사용자는 어떤 화면에서 무엇을 해야 하는가?
- 💭 어떤 버튼을 누르면 어디로 이동하는가?
- 💭 어떤 데이터가 어느 시점에 노출되는가?
- 💭 기능 간 관계는 어떻게 연결되는가?

이 질문들을 텍스트로 정리해 Cursor에 전달했고,  
그 자체가 **프롬프트의 기반**이 되었습니다.

### 🧠 프롬프트 작성 패턴

#### Level 1 — 단순 기능 요청

```
케이스 내용을 구조적으로 정리하기 위해 에디터에 리스트 기능을 추가하고 싶습니다.
번호 리스트와 불렛 리스트 모두 사용할 수 있어야 하고,
생성·편집 화면 모두 동일한 동작을 지원하면 좋겠습니다.
엔터 시 다음 항목 자동 추가, 리스트 해제 기능도 포함해주세요.
```

#### Level 2 — Flow 중심 상세 프롬프트

```
[화면 구성]
- 왼쪽: 폴더 트리
- 오른쪽: 케이스 목록

[동작 흐름]
1. 폴더 클릭 시 해당 케이스 필터링
2. 케이스 클릭 시 상세 정보 표시
3. 드래그앤드롭으로 폴더 이동
4. 순환 참조 방지

[기능 연계]
- 폴더 삭제 시 하위 케이스 삭제
- 이동 시 실시간 업데이트
```

### 결론

AI가 제시한 기술을 그대로 따르되,  
**흐름과 목적은 제가 직접 설계**했습니다.

## 🐞 디버깅 전략 — QA 경험의 빛

문제가 생겼을 때 **QA 경험**은 큰 도움이 되었습니다.

### 1️⃣ 로그 추가 요청

```
각 단계마다 console.log를 추가해서 데이터가 언제 변경되는지 확인해주세요.
```

### 2️⃣ 논리적 추적

```
폴더 삭제 시 DB의 CASCADE 옵션이 적용되는지 확인해주세요.
삭제 후 목록을 다시 가져오고 있는지도 확인해주세요.
```

### 3️⃣ 집요한 원인 추적

의심되는 흐름을 끝까지 따라가면  
의외로 문제의 원인을 발견할 때가 많았습니다.

### 4️⃣ AI의 맥락 유지 한계 보완

- ✅ 문제를 구체적으로 설명
- ✅ 기능 단위로 흐름을 재정리
- ✅ 현상을 객관적으로 전달

이 과정을 반복하며 **안정성을 확보**해 나갔습니다.

## 🏁 완성된 시스템 — TMS

2025년 12월,  
초기 기획했던 MVP 기능을 모두 갖춘 **TMS의 첫 버전**이 완성되었습니다.

현재는 **실무에서도 적용 중**입니다.

### 🎯 주요 기능

#### ✔️ 로그인 / 권한 관리

- 가입 후 관리자 승인 필요
- 비밀번호 초기화
- JWT Access Token 인증

#### ✔️ CSV Import

- 동일 헤더 자동 매핑
- Import 화면에서 폴더 생성 가능

#### ✔️ 테스트 케이스 관리

- 계층형 폴더 구조
- 드래그앤드롭 이동
- Rich Text Editor 기반 케이스 작성

#### ✔️ 플랜 생성 및 관리

- 폴더 기반 테스트 플랜 생성
- 진행률 집계
- 실시간 실행 모니터링

#### ✔️ 테스트 실행

- 개별/벌크 결과 입력
- 메모에 URL 입력 시 자동 링크 처리

## 🌱 얻은 것들

### 1️⃣ 즉각적 유지보수 가능

외부 벤더 패치를 기다릴 필요 없이 **직접 수정 가능**

### 2️⃣ 빠른 기능 추가

"이런 기능 있었으면 좋겠다" 싶은 건 **직접 구현 가능**

### 3️⃣ 성능 개선

| 항목 | 응답 시간 | 비고 |
|------|----------|------|
| TestRail API | 초 단위 | 느린 응답 |
| TMS API | **ms 단위** | 실무에서 체감 차이가 큼 |

### 4️⃣ 비용 절감

팀에 딱 맞는 시스템을 **자체 운영 가능**

## 📚 배운 것들 — 시스템 이해력의 성장

### 가장 큰 깨달음

> **"언어를 몰라도, 기능을 정확히 이해하면 AI는 따라온다."**

### 성장한 역량

이 프로젝트를 통해 다음 역량이 크게 성장했습니다:

- 🎯 기능 목적을 언어로 정확히 정의하는 능력
- 🎯 흐름과 구조를 설명하는 능력
- 🎯 문제를 논리적으로 분석하는 능력
- 🎯 시스템 이해 기반의 설계 능력

### 중요한 깨달음

AI가 코드를 쓰지만,  
**"코드가 왜 이렇게 동작하는지 이해하는 것"**은 사람이 해야 했습니다.

## 🧭 회고 — AI는 도구이고, 방향은 사람이 정한다

AI를 활용하면 개발이 쉬워질 거라 생각했지만,

> **"설계가 명확하지 않으면 AI는 결코 제대로 구현하지 못한다."**

### 마주한 한계

기능이 늘어날수록 다음과 같은 문제가 더 자주 발생했습니다:

- 🚨 맥락 손실
- 🚨 기존 기능의 의도치 않은 변경
- 🚨 기능 간 연계 오류

### 얻은 성장

하지만 이 과정은 오히려 다음 역량이 **폭발적으로 성장**한 값진 경험이었습니다:

- 💡 흐름 설계 능력

- 💡 시스템 사고
